# План миграции (Migration Roadmap)

## Цель: Постепенный отказ от JS Serverless (Vercel) в пользу Go (Backend) + Очереди

| Фаза | Описание | Срок | Риск |
| :--- | :--- | :--- | :--- |
| **Фаза 1: Инфраструктура** | Поднятие скаффолдинга Go, настройка CI/CD, Docker Compose для локальной разработки, подключение `sqlc` для БД. Написание Makefile. | 1 неделя | Низкий |
| **Фаза 2: AI API Gateway** | Миграция `lib/openRouter.js` и вызовов AssemblyAI в пакет `internal/ai/` на Go с поддержкой Retry/Timeout. | 2 недели | Средний |
| **Фаза 3: Транскрибация и Очереди** | Перенос `api/transcribe.js` в Go с использованием `Asynq`. Создание воркеров для загрузки видео с Instagram и вызова AssemblyAI. | 2 недели | Высокий (зависит от стабильности очередей) |
| **Фаза 4: Scriptwriter (Бизнес-логика)** | Разбор God-объекта `api/scriptwriter.js` (80КБ) на отдельные сервисы в Go: анализ структуры, генерация хуков, анализ каруселей. | 3-4 недели | Очень высокий (много логики парсинга JSON из LLM) |
| **Фаза 5: Фронтенд и Отключение JS** | Обновление запросов на фронтенде для работы с Go API вместо Vercel. Удаление папки `api/`. | 1 неделя | Низкий (чистая подмена эндпоинтов) |

## Что остаётся на TS/JS (до конца Фазы 5)
Вся папка `src/` (Фронтенд), а также существующие миграции и Edge-функции Supabase (папка `supabase/`).

## Критерии успеха:
1. Отсутствие таймаутов на Vercel (задачи выполняются асинхронно в Go-воркерах).
2. Строгая типизация запросов к БД и AI-сервисам.
3. Уменьшение дублирования кода.
